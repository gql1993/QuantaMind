# QuantaMind 研发路线图（多智能体协作）

与《QuantaMind量智大脑详细设计说明书》§13.4 实施阶段及《QuantaMind产品原型说明》对齐；各阶段由架构师 → 研发工程师 → 测试工程师 按序或并行协作。

---

## 阶段 0：基础与多智能体流程就绪（当前）

| 角色 | 任务 | 交付物 | 验收 |
|------|------|--------|------|
| 架构师 | 确认 Gateway/Brain/Hands/Memory 分层与现有 API 契约；补充“任务列表”API 契约（路径、请求/响应） | 接口说明或 ADR | 与详细设计 §11、§3.2 一致 |
| 研发工程师 | 实现 `GET /api/v1/tasks`、`GET /api/v1/tasks/{id}`（内存版）；共享模型沿用 `TaskInfo`/`TaskStatus` | 代码、API 可调 | 接口返回符合契约 |
| 测试工程师 | 编写任务 API 契约测试与健康检查测试；可 pytest 执行 | 测试计划摘要、自动化用例 | 全部通过 |

**产出**：任务列表 API 可用；多智能体流程与 `dev/agents/`、`dev/ROADMAP.md` 就绪。

---

## 阶段 1：对话 + 任务页与客户端主流程（P0）

| 角色 | 任务 | 交付物 | 验收 |
|------|------|--------|------|
| 架构师 | 会话/对话/任务与客户端的完整数据流；WebSocket 与 REST 使用场景；审批流接口契约 | ADR、接口补充 | 客户端可仅凭 Gateway API 完成对话与任务列表 |
| 研发工程师 | 1）桌面/Web 客户端：主导航（对话/任务/自主发现/技能/设置）+ 对话页（会话列表、消息流、输入框）；2）任务页：Tab（全部/我发起的/待我审批/已完成）、列表与详情占位；3）任务 API 与会话绑定（如 task 带 session_id） | 客户端页面、Gateway 小改 | 原型五块可切换；对话可发收；任务列表可展示 |
| 测试工程师 | 对话 API 流式/非流式契约测试；任务 API 契约与列表场景；客户端 E2E：打开对话、发一条消息、切到任务页 | 用例、自动化、报告 | 关键路径通过 |

**产出**：用户可在一个客户端内完成对话与任务列表查看，与产品原型 P0 一致。

## 阶段 2：自主发现与技能页（P1）

| 角色 | 任务 | 交付物 | 验收 |
|------|------|--------|------|
| 架构师 | Heartbeat 发现列表、技能列表与 Gateway 的接口约定；`/api/v1/heartbeat/discoveries`、`/api/v1/skills` 响应格式 | 接口规范 | 与详细设计 §7、§9 一致 |
| 研发工程师 | 1）Gateway：`/api/v1/heartbeat/discoveries`（可 Mock）；2）客户端：自主发现页（筛选、卡片列表、“在对话中继续”）；技能页（列表、触发词、一键触发占位） | 接口 + 客户端页面 | 自主发现/技能页与原型一致 |
| 测试工程师 | 发现列表与技能列表 API 契约测试；客户端自主发现/技能页基本交互 | 用例、自动化 | 通过 |

**产出**：自主发现与技能两块在客户端可用，数据可由 Mock 支撑。

## 阶段 3：审批流与 Hands 工具调用展示（P0 深化）

| 角色 | 任务 | 交付物 | 验收 |
|------|------|--------|------|
| 架构师 | 审批 API（通过/拒绝）、任务状态与“待审批”来源；Hands 工具调用与对话中展示的契约 | ADR、接口 | 与详细设计 §14、§15 数据流一致 |
| 研发工程师 | 1）Gateway：审批端点；任务创建/更新（含待审批状态）；2）对话中工具调用与待审批卡片展示；任务页审批操作 | 代码、客户端 | 用户可在对话或任务页完成审批 |
| 测试工程师 | 审批 API 契约与状态机；E2E：发起需审批任务 → 审批 → 任务完成 | 用例、自动化 | 通过 |

**产出**：完整审批流与工具调用在对话中的展示。

## 阶段 4：与平台/数据中台对接（Mock → 真实）

| 角色 | 任务 | 交付物 | 验收 |
|------|------|--------|------|
| 架构师 | Q-EDA/MES/测控/数据中台对接的模块边界与接口契约复核；Mock 与真实切换策略 | ADR、契约对照表 | 与 §5.7–§5.11 一致 |
| 研发工程师 | Hands 适配器：Q-EDA/MES/测控/数据中台 Mock 实现；可配置切换为真实端点；Memory 与数据中台查询 Mock | 适配器、配置、文档 | Mock 下端到端可用；配置切换无代码改动 |
| 测试工程师 | Mock 环境下集成测试；真实端点联调检查清单与冒烟用例 | 用例、报告、检查清单 | Mock 全绿；真实环境有清单可执行 |

**产出**：平台与数据中台可由 Mock 支撑开发与测试，具备切换真实环境能力。

## 阶段 5：Heartbeat 多级与自主发现真实化（P1 深化）

| 角色 | 任务 | 交付物 | 验收 |
|------|------|--------|------|
| 架构师 | Heartbeat 多级调度与发现写入 Memory/API 的接口；与数据中台/文献等数据源的契约 | 接口与数据流说明 | 与 §7、§15 一致 |
| 研发工程师 | Heartbeat 多级调度实现；发现持久化与 `/api/v1/heartbeat/discoveries` 真实数据；可选文献/实验建议 Mock | 代码、配置 | 发现列表有真实来源 |
| 测试工程师 | Heartbeat 调度与发现 API 的集成测试；发现类型与筛选 | 用例、自动化 | 通过 |

**产出**：自主发现由 Heartbeat 真实产出，可筛选与跳转对话。

---

## 使用说明

- **执行顺序**：建议按阶段 0 → 1 → 2 → 3 → 4 → 5 推进；阶段内“架构师 → 研发 → 测试”可串行，或架构师定稿后研发与测试并行。
- **智能体使用**：每个阶段开始时，将本路线图对应阶段表格 + `dev/agents/` 中对应角色指令 + 详细设计/原型 注入该角色会话，由该智能体产出交付物；再交给下一角色或下一阶段。
- **与详细设计对应**：§11 接口规范、§14 业务流、§15 数据流为权威参考；若路线图与说明书冲突，以说明书为准并更新本路线图。
