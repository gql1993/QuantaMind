## 2. QuantaMind 的呈现形态：客户端

**QuantaMind 的表现方式是客户端。** 用户感知到的「QuantaMind」是一套可安装、可打开的**客户端应用**，而不是直接操作服务器或 API。背后的 AI 中台（Gateway、Brain、Hands、Memory、Heartbeat、Skills）作为服务端能力层，仅通过客户端对外呈现。

### 2.1 客户端优先架构

```

│  │  · 嵌入式面板 (Q-EDA 内嵌)       — 在设计工具内直接调起      │    │
```

### 2.2 客户端形态说明

| 形态 | 适用场景 | 说明 |
|------|----------|------|
| **桌面客户端** | 日常科研、全功能使用 | QuantaMind的**主表现形态**。独立安装包，集成对话、任务、知识、心跳状态、设计加速等全部能力，支持离线/内网。 |
| **Web 客户端** | 轻量查看、审批、看板 | 浏览器访问内网地址，用于查看自主发现、审批任务、看板与报表，不替代桌面端的深度操作。 |
| **嵌入式面板** | 设计/仿真时即用即调 | 在 Q-EDA 等工具内嵌「QuantaMind」面板，在设计界面内直接与 AI 对话、调起设计加速，无需切窗口。 |
| **CLI** | 脚本、CI、自动化 | 命令行调用，用于流水线、自动化脚本，仍视为一种「客户端」入口。 |

### 2.3 客户端与服务端职责划分

| 职责 | 客户端 | 服务端 |
|------|--------|--------|
| **表现与交互** | 全部（UI、对话、看板、审批、设置） | 无直接 UI |
| **会话与认证** | 发起登录、维持会话、展示权限 | 鉴权、会话管理、权限校验 |
| **业务逻辑** | 无（或仅本地缓存/离线草稿） | 全部（路由、推理、工具调用、心跳） |
| **数据持久化** | 可选本地缓存、用户偏好 | 记忆、知识图谱、向量库、数据仓库 |
| **与六大平台交互** | 不直接调用 | 通过 QuantumHands 统一调用 |

**结论：** QuantaMind = **以客户端为表现方式** 的量子科研 AI 中台；服务端是能力层，客户端是唯一面向用户的呈现层。

| OpenClaw 特性 | 对QuantaMind的价值 |
|--------------|----------------|
| **本地优先** | 量子科研数据高度敏感，全部本地运行确保数据安全 |
| **自主运行（Heartbeat）** | 实现 AI 科学家 7×24 小时自主科研探索 |
| **模型无关** | 灵活切换国产大模型（通义千问/DeepSeek）与国际模型 |
| **技能扩展（Skills）** | 量子科研各环节封装为可复用技能模块 |
| **Gateway 统一编排** | 单一进程管理所有 Agent，简化部署运维 |
| **Markdown 记忆** | 科研知识以人类可读格式持久化，便于审核与迭代 |
| **开源社区** | 182K+ Stars，生态活跃，持续演进 |

## 3. 核心设计理念：OpenClaw → QuantumClaw

将 OpenClaw 的通用 Agent 架构**领域特化**为面向量子科技的自主科研系统：

```
┌─────────────────────────────────────────────────────────────────────┐
│                    OpenClaw 通用架构                                  │
│                                                                     │
│   Gateway ──── Brain ──── Hands ──── Memory ──── Heartbeat          │
│      │           │          │          │            │                │
│   ───┼───────────┼──────────┼──────────┼────────────┼───── 领域特化  │
│      │           │          │          │            │                │
│      ▼           ▼          ▼          ▼            ▼                │
│                    QuantumClaw 量子科研架构                           │
│                                                                     │
│   Quantum     Quantum    Quantum    Quantum      Quantum            │
│   Gateway     Brain      Hands      Memory       Heartbeat          │
│   (中枢网关)  (多模型     (六大平台   (知识图谱    (自主科研           │
│              推理引擎)    工具层)    +实验记忆)    心跳引擎)           │
│                                                                     │
│              + QuantumSkills (量子科研技能市场)                        │
│              + QuantumAgents (AI科学家智能体团队)                      │
└─────────────────────────────────────────────────────────────────────┘
```

**关键差异化设计：**

1. **Brain 升级为 MoE（混合专家）路由引擎**：不再是单一 LLM，而是 4 大垂直领域大模型 + 专家模型库的协同推理
2. **Hands 扩展为六大平台工具接口**：从通用 Shell/Browser 扩展为 Q-EDA/MES/仿真/测控等专业工具的 API 调用
3. **Memory 升级为量子知识图谱**：从简单 Markdown 笔记扩展为结构化知识图谱 + 向量检索 + 实验日志
4. **Heartbeat 升级为科研心跳**：从定时任务检查升级为多层级自主科研循环（文献监控/实验设计/假设验证）
5. **Skills 特化为量子科研技能**：封装芯片设计、仿真分析、工艺优化等领域专有技能

## 5. QuantumClaw Gateway — 量子科研中枢网关

### 5.1 架构映射

| OpenClaw Gateway | QuantumClaw Gateway |
|------------------|---------------------|
| WebSocket 控制平面 | 量子科研 WebSocket 控制平面 + REST API |
| 通道桥接（WhatsApp/Slack等） | 通道桥接（Q-EDA面板/Web/CLI/飞书/Notebook） |
| 路由器（消息→处理器） | **MoE 路由器**（任务→专家Agent） |
| 认证会话管理 | **多租户科研项目隔离** + RBAC 权限 |

### 5.2 核心组件

```yaml
# QuantumClaw Gateway 配置示例
gateway:
  host: 0.0.0.0
  port: 18789
  
  channels:
    - type: qeda_panel        # Q-EDA 集成面板
      endpoint: /ws/qeda
    - type: web_dashboard      # Web 研发看板
      endpoint: /ws/dashboard
    - type: feishu             # 飞书机器人
      app_id: ${FEISHU_APP_ID}
    - type: cli                # 命令行交互
      endpoint: /ws/cli
    - type: jupyter_kernel     # Jupyter Notebook 内核
      endpoint: /ws/notebook

  router:
    type: moe_quantum          # MoE 混合专家路由
    strategies:
      - intent_classification  # 意图分类
      - agent_affinity         # Agent 亲和性
      - load_balancing         # 负载均衡

  session:
    persistence: redis
    project_isolation: true    # 科研项目隔离
    max_concurrent_agents: 10
```

### 5.3 MoE 智能路由器

路由器借鉴 MoE（Mixture of Experts）思想，根据任务语义自动分发至最合适的专家Agent：

```
用户输入 → 意图分类 → 路由决策 → Agent 调度
              │
              ├── "设计一个5比特transmon芯片"  → AI芯片设计师Agent
              ├── "分析这个材料的超导临界温度"  → AI材料科学家Agent
              ├── "优化约瑟夫森结工艺参数"     → AI工艺工程师Agent
```

### 6.3 推理协同机制

当任务需要多个模型协同时，采用**链式推理 (Chain)** 或**并行推理 + 融合 (Parallel + Merge)**：

```
场景：设计一个新型高保真度Transmon量子比特

步骤1 [理论物理模型]: 计算目标频率/非谐性的理论约束
    → 输出: {freq: 5.0-5.5 GHz, anharmonicity: -300±20 MHz}
    
步骤2 [材料模型 + 专家模型] (并行):
    材料模型: 推荐最优衬底材料和超导薄膜
    频率预测器: 反向求解满足约束的几何参数
    → 输出: {substrate: sapphire, film: Al, pad_width: 320μm, ...}

步骤3 [工艺模型]: 评估可制造性，推荐工艺窗口
    → 输出: {feasible: true, jj_area: 0.04μm², process_window: {...}}
    
步骤4 [视觉模型]: 预审版图，检查潜在设计违规
    → 输出: {violations: 0, suggestions: ["增大接地间距到10μm"]}

融合输出 → 完整设计方案 + 置信度评分
```

```yaml
# Transmon 量子比特设计技能

## 触发条件
当用户请求设计 Transmon 类型量子比特时触发。

## 执行流程

### Step 1: 需求澄清
向用户确认以下设计参数:
- 目标频率范围 (默认 4.5-5.5 GHz)
- 非谐性目标 (默认 -300 ± 30 MHz)
- 耦合方式 (电容/电感/可调)
- 读出方式 (色散读出/JPA增强)

### Step 2: 理论计算
调用 theorist_agent 进行:
- 基于目标频率反算 EJ/EC 比例
- 计算约瑟夫森结面积估算值
- 评估非谐性与频率的 trade-off

### Step 3: 几何参数生成
调用 frequency_predictor 专家模型:
- 输入: {target_freq, anharmonicity, coupling_type}
- 输出: {pad_width, pad_height, pad_gap, jj_area, claw_length}

### Step 4: 版图生成
调用 qeda_create_design 和 qeda_add_qubit:
- 创建设计项目
- 放置 Transmon 组件
- 配置几何参数

### Step 5: 设计检查
调用 qeda_check_drc:
- 最小间距检查
- 接地平面连续性
- 金属密度均匀性

### Step 6: 知识沉淀
将设计参数和决策依据记录到项目记忆
```
